# Архитектура отдельной `Homey` app для `VARMANN / Vartronic`

Дата: `2026-04-15`

Статус: проектный документ по целевой интеграции `VARMANN / Vartronic` в `Homey Pro` без внешнего poller.

## 1. Зачем нужен этот документ

Этот документ фиксирует целевую архитектуру отдельной `Homey` app для конвекторов `VARMANN / Vartronic`.

Он отвечает на вопросы:

- можно ли сделать отдельную app внутри `Homey`, а не через общий `Modbus` app;
- будет ли такая app сама добавлять устройства;
- где будет жить логика `autopoll / heartbeat`;
- как это будет выглядеть в интерфейсе `Homey`;
- какие есть ограничения, риски и условия для продакшн-использования.

## 2. Короткий вывод

Да, отдельная `Homey` app под `VARMANN / Vartronic` возможна и архитектурно выглядит чище, чем связка:

- generic `Modbus` app;
- `Flows`;
- `HomeyScript`;
- ручной heartbeat.

В такой схеме:

- app живет внутри `Homey Pro`;
- сама подключается по `LAN` к `Modbus TCP` шлюзу, например `192.168.1.8:502`;
- сама знает список устройств `Modbus ID 16..23`;
- сама делает циклический `polling`;
- сама обновляет состояние устройств в интерфейсе `Homey`;
- сама пишет команды в нужные регистры.

Для пользователя это будет выглядеть как набор обычных климатических устройств в `Homey`, а не как набор сырых `Modbus`-регистров.

## 3. Почему это вообще возможно в `Homey`

По официальной документации `Homey Apps SDK`:

- `Homey` app запускается локально на самом `Homey`, как `Node.js` или `Python` bundle;
- app может иметь свои `drivers` и `devices`;
- driver может использовать `connectivity: ["lan"]`;
- `Homey` может обращаться к устройствам в локальной сети без дополнительных permissions;
- pairing устройств реализуется через встроенные или custom pairing views;
- device-класс может:
  - принимать действия пользователя через `registerCapabilityListener()`;
  - обновлять состояние в UI через `setCapabilityValue()`;
- app может иметь собственные `Flow cards`, если стандартных действий capabilities окажется недостаточно.

Практический вывод:

- отдельная `LAN`-app под `VARMANN` не противоречит модели `Homey SDK`;
- heartbeat вполне может жить внутри app;
- устройства могут добавляться не через стандартный `Modbus` app, а через собственный driver этой app.

## 4. Что именно должна решать app

Эта app нужна не для всего `Modbus` в доме, а только для `VARMANN / Vartronic`.

Причины:

- у `VARMANN` есть обязательный циклический обмен;
- generic `Modbus` app в `Homey` не является специализированным слоем для heartbeat;
- `Haier` и `VARMANN` нельзя смешивать на одной шине;
- `Haier` и `VARMANN` имеют разные serial-профили:
  - `Haier`: `9600 8N2`;
  - `VARMANN`: `9600 8N1`.

Значит app под `VARMANN` должна быть:

- специализированной;
- локальной;
- ориентированной на один `Modbus TCP` endpoint и набор `Modbus ID`;
- ответственной за heartbeat и resync после потери связи.

## 5. Где живет app и как идет трафик

### 5.1. Где выполняется код

Код app живет и запускается внутри `Homey Pro`.

Это не `Raspberry Pi`, не внешний сервис и не `HomeyScript`.

Схема такая:

```text
Homey app (внутри Homey Pro)
  -> LAN / TCP
  -> USR gateway 192.168.1.8:502
  -> Modbus ID 16..23
```

### 5.2. Что такое устройство в интерфейсе `Homey`

В интерфейсе `Homey` пользователь будет видеть не `Modbus Device`, а собственные устройства app, например:

- `Конвектор 16`;
- `Конвектор 17`;
- `Конвектор 18`;
- или уже по комнатам:
  - `Гостиная`;
  - `Спальня`;
  - `Кабинет`.

## 6. Как app будет выглядеть для пользователя

### 6.1. В списке устройств

Пользователь увидит набор обычных климатических устройств.

Они должны быть описаны как устройства класса уровня:

- `thermostat`, если хотим привычную климатическую модель;
- либо другой близкий класс, если тестирование покажет, что стандартный thermostat UI неудобен.

### 6.2. В карточке одного устройства

Минимально полезный интерфейс одного конвектора:

- текущая температура;
- целевая температура;
- режим;
- скорость вентилятора;
- аварийный статус;
- статус доступности.

Дополнительно, но не обязательно для MVP:

- `last seen`;
- текущий `Modbus ID`;
- диагностический статус heartbeat;
- версия профиля / драйвера.

### 6.3. Во `Flows`

Пользователь должен работать не с регистрами, а с действиями уровня устройства:

- установить температуру;
- включить нагрев;
- сменить режим;
- выключить вентилятор;
- проверить аварийный статус;
- проверить доступность устройства.

## 7. Как будет устроен pairing

### 7.1. Общая идея

Устройства будут добавляться через собственный driver app.

Есть два разумных сценария pairing:

1. `Ручной ввод endpoint + выбор ID`
2. `Ввод endpoint + массовое добавление диапазона ID`

Для этого проекта более практичен второй вариант.

### 7.2. Предлагаемый pairing для MVP

Во время pairing пользователь:

1. вводит `IP` шлюза, например `192.168.1.8`;
2. вводит `port`, обычно `502`;
3. указывает диапазон `Modbus ID`, например `16..23`;
4. app пробует прочитать безопасный регистр у каждого `ID`;
5. app показывает список найденных устройств;
6. пользователь подтверждает, какие из них добавить в `Homey`.

На выходе в `Homey` появляются отдельные devices:

- один device = один `VARMANN` controller;
- у каждого device в `data/store/settings` лежат:
  - `host`;
  - `port`;
  - `modbusId`;
  - возможно имя комнаты или alias.

### 7.3. Почему не надо завязываться только на app settings

Документация `Homey` допускает persistent app settings, но прямо указывает, что данные, необходимые устройству для работы, лучше получать при pairing.

Практический вывод:

- глобальные app settings можно оставить только для общих defaults;
- реальные `host/port/id` лучше привязывать к device во время pairing.

## 8. Как будет устроен autopoll / heartbeat

### 8.1. Где он живет

Autopoll живет внутри app.

Не в:

- `Flow`;
- `Advanced Flow`;
- `HomeyScript`;
- `USR` heartbeat packet.

А именно в runtime самой app.

### 8.2. Базовый принцип работы

Внутри app есть scheduler или async loop, который:

1. берет список активных `VARMANN` устройств;
2. выстраивает их в очередь;
3. по кругу делает запросы к `192.168.1.8:502`;
4. читает безопасные регистры;
5. обновляет состояние device в `Homey`;
6. повторяет цикл.

Схема:

```text
poll cycle:
  ID 16 -> read
  ID 17 -> read
  ID 18 -> read
  ...
  ID 23 -> read
  repeat
```

### 8.3. Какой транспорт должен быть в app

Целевой вариант:

- один shared transport manager на endpoint `host:port`;
- одна очередь запросов;
- сериализованный обмен;
- reconnect при ошибке;
- backoff при повторных сбоях.

Практический смысл:

- не плодить независимые TCP-клиенты на каждое устройство;
- не создавать гонки по одному `Modbus TCP` шлюзу;
- централизованно управлять нагрузкой и логированием.

### 8.4. Какие регистры нужны для heartbeat

Минимальный безопасный heartbeat можно строить на чтении одного безопасного регистра, например:

- `0x000F Alarm`

Но для реальной UI-синхронизации полезнее читать небольшой минимальный набор:

- `0x0009 TmpOut`
- `0x000F Alarm`
- `0x0003 HeatChill`
- `0x0006 UstFan`

Конкретный минимальный пул регистров нужно уточнить по живому поведению устройства и по итоговой карте регистров.

### 8.5. Какой интервал polling нужен

Документально подтверждено, что:

- у `VARMANN` есть `TimeLan`;
- по умолчанию он равен `2 s`;
- если обмен пропадает, устройство может уйти в ручной режим.

Из этого следуют два режима проектирования:

1. `Строгий режим`
Если оставляем `TimeLan` по умолчанию, polling должен быть очень частым и дисциплинированным.

2. `Ослабленный режим`
Если на объекте подтверждается безопасное увеличение `TimeLan`, app получает больший временной запас и проще переживает краткие паузы и рестарты.

Для проектного документа это означает:

- MVP не должен молча предполагать, что дефолтный `2 s` приемлем;
- app должна иметь отдельную конфигурацию polling profile;
- обязательно нужен lab-тест с реальным `VARMANN` и реальным шлюзом.

## 9. Как будут обрабатываться команды пользователя

### 9.1. Общая схема

Когда пользователь меняет состояние в `Homey`, device-класс app:

1. получает изменение capability;
2. преобразует его в `Modbus`-операцию;
3. отправляет запрос через shared transport manager;
4. при успехе обновляет локальное состояние;
5. при ошибке ставит device в degraded или unavailable state.

### 9.2. Какие регистры нужны в первой версии

Для MVP достаточно базовых рабочих регистров:

- `0x0002 UstTmp` — уставка температуры;
- `0x0003 HeatChill` — режим;
- `0x0006 UstFan` — уставка скорости вентилятора;
- `0x000F Alarm` — авария;
- `0x0009 TmpOut` — текущая температура.

## 10. Что должно происходить при перезагрузке `Homey` или app

Это один из важнейших разделов всей архитектуры.

Если `Homey` перезагрузится или app временно упадет:

- heartbeat остановится;
- `VARMANN` может уйти в ручной режим;
- после восстановления app должна не просто переподключиться, а сделать `resync`.

Нормальная последовательность после reconnect:

1. поднять transport;
2. прочитать минимальное состояние устройства;
3. обновить capabilities;
4. сравнить фактическое состояние с последним желаемым;
5. при необходимости повторно отправить целевые параметры;
6. вернуться в обычный polling loop.

Из этого следует, что app должна хранить:

- last known actual state;
- last desired state;
- timestamp последней успешной связи.

## 11. Что именно надо хранить в app

### 11.1. На уровне app

Общий уровень app:

- менеджер транспортов по `host:port`;
- конфигурация polling profile;
- глобальные defaults для pairing;
- логика reconnect и очереди.

### 11.2. На уровне device

На device:

- `host`;
- `port`;
- `modbusId`;
- display name;
- last actual state;
- last desired state;
- last seen;
- состояние доступности.

## 12. Риски и ограничения

### 12.1. Главный технический риск

Главный риск не в UI и не в pairing, а в устойчивом heartbeat при реальной нагрузке и реальном `TimeLan`.

То есть сложная часть проекта:

- не показать плитку в `Homey`;
- а удержать `VARMANN` в сетевом режиме без срывов.

### 12.2. Риск публикации в `Homey App Store`

Официальная документация рекомендует discovery для `LAN`-устройств и указывает, что app, требующие ручного ввода IP там, где discovery мог бы использоваться, могут быть отклонены при публикации.

Практический вывод:

- для локальной private app, устанавливаемой через CLI, это не блокер;
- для публичной app придется отдельно думать о discovery или обосновании ручного pairing.

### 12.3. Ограничение по платформе

Эта app должна проектироваться под `Homey Pro local`, а не под `Homey Cloud`.

Причины:

- нужен локальный `LAN` доступ;
- нужны custom settings / pairing flexibility;
- heartbeat должен жить локально рядом с endpoint.

### 12.4. Ограничение по охвату

Эта архитектура не должна пытаться сразу стать универсальной `Modbus`-платформой для всего дома.

Правильный scope первой версии:

- только `VARMANN / Vartronic`;
- только их отдельная шина;
- только их endpoint `host:port`;
- только их карта регистров.

## 13. Почему это чище, чем `Modbus app + HomeyScript`

Плюсы отдельной app:

- один владелец polling и reconnect;
- нет зависимости от flow card semantics стороннего app;
- нет ручного heartbeat через `Flows`;
- нет расползания логики между `Modbus` app, `HomeyScript` и `Advanced Flow`;
- устройства выглядят как нормальные климатические devices;
- проще масштабировать и отлаживать.

Минусы отдельной app:

- это уже разработка, а не только настройка;
- нужно самим реализовать `Modbus TCP` клиент и transport queue;
- нужно самим тестировать reconnect и heartbeat.

Итог:

- как временный workaround generic `Modbus` app еще допустим;
- как постоянное решение для `VARMANN` отдельная app выглядит правильнее.

## 14. Предлагаемая структура MVP

### 14.1. Объем MVP

MVP первой версии должен уметь:

- добавлять конвекторы `VARMANN` через pairing;
- хранить `host`, `port`, `modbusId`;
- читать минимальный набор регистров;
- обновлять состояние устройства в `Homey`;
- принимать базовые команды пользователя;
- держать heartbeat;
- корректно восстанавливаться после reconnect.

### 14.2. Что войдет в MVP

- один driver `varmann_convector`;
- device class для одного конвектора;
- shared transport manager;
- poll scheduler;
- mapping регистров в capabilities;
- базовый reconnect;
- диагностический лог.

### 14.3. Что не надо тащить в первый релиз

Не нужно в MVP:

- универсальный `Modbus` abstraction layer для всех производителей;
- `Haier` поддержку;
- сложный multi-endpoint UI;
- продвинутые графики;
- попытку охватить все возможные варианты fan/mode до live validation.

## 15. Рекомендуемый план разработки

### Этап 1. Lab MVP

1. Поднять app локально на `Homey Pro`.
2. Подключить один `VARMANN` через `USR` gateway.
3. Проверить pairing одного device.
4. Проверить polling и reconnect.
5. Проверить, что устройство не уходит в ручной режим.

### Этап 2. Масштабирование на диапазон

1. Добавить `ID 16..23`.
2. Проверить round-robin polling.
3. Проверить полный цикл heartbeat под рабочей нагрузкой.
4. Проверить resync после рестарта `Homey`.

### Этап 3. UI и Flows

1. Уточнить capabilities.
2. Доработать карточки устройств.
3. Добавить пользовательские Flow actions при необходимости.
4. Довести UX до состояния "обычные климатические девайсы".

## 16. Открытые вопросы

Перед реализацией нужно закрыть следующие вопросы:

- какой безопасный реальный polling interval допустим на объекте;
- нужно ли менять `TimeLan` на всех контроллерах заранее;
- какой минимальный набор регистров нужен для надежного heartbeat;
- какая модель UI лучше для `Homey`:
  - `thermostat`;
  - или custom capability composition;
- нужен ли один endpoint на все устройства навсегда, или архитектуру сразу делать на несколько endpoint;
- насколько предсказуемо `VARMANN` восстанавливает сетевой режим после reconnect без явной повторной записи параметров.

## 17. Итоговое решение

Для этого проекта целевая архитектура выглядит так:

- отдельная `Homey Pro` app;
- scope только `VARMANN / Vartronic`;
- devices добавляются через собственный pairing;
- heartbeat и autopoll живут внутри app;
- пользователь видит нормальные климатические devices;
- generic `Modbus` app и `HomeyScript` остаются только временным workaround, а не целевой архитектурой.

Это решение не является "универсальной платформой для всего `Modbus` дома", но именно поэтому оно выглядит реалистичным, устойчивым и инженерно аккуратным для текущего объекта.
